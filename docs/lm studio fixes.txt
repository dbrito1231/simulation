Here they are, ordered by expected impact.

1. The thinking-disable field is almost certainly a no-op (biggest win)
server.py:2733 sends "thinking": {"type": "disabled", "budget_tokens": 0} — but that's Anthropic API format. LM Studio's OpenAI-compatible endpoint silently ignores unknown fields. The evidence it's not working is already in your own docs: lms_config.md line 62 notes "responses often put text in reasoning_content with empty content" — meaning the model is still reasoning on every routine villager turn. That's likely where most of the 18s-median / 22s-p90 decision latency goes.

Fixes to try, in order:

Qwen's soft switch: append /no_think to the user prompt (or system prompt) for routine turns.
"chat_template_kwargs": {"enable_thinking": false} in the payload — recent LM Studio builds pass this through to the chat template.
LM Studio's per-model settings UI (My Models → qwen3.5 → default reasoning off), if the build exposes it.
Then verify in lm_studio.jsonl: content populated, reasoning_content empty, latency drop. Keep thinking ON for the MODEL_SMART path (elder, invention, sprite turns) — that gives you real quality differentiation between the smart/fast tiers even with a single loaded model, and invention already has the 75s timeout to absorb it. Bonus: with reasoning off, max_tokens: 512 becomes genuinely roomy; right now routine turns risk the same finish_reason: "length" mid-JSON truncation you already diagnosed and fixed for invention turns (worth grepping lm_studio.jsonl for "length" on non-invention calls to confirm).

2. Pin the sampling parameters — only temperature is set
build_decision_payload sends only temperature; top_k/top_p/min_p fall back to whatever LM Studio's preset happens to be, which drifts across app updates and model reloads (the same drift problem the config file already warns about for context/parallel). Pin Qwen's recommended values explicitly in the payload:

Non-thinking turns: top_p: 0.8, top_k: 20, min_p: 0
Thinking turns (invention/elder): top_p: 0.95, top_k: 20
Also worth an experiment: a modest presence_penalty (Qwen tolerates up to ~1.5) on routine turns could further cut the move_to_district fixation you benchmarked at 32%.

3. Speculative decoding (free ~1.5–2× decode speedup)
LM Studio supports draft-model speculative decoding. Load a tiny same-family draft (e.g. a qwen3.5 0.5–1B quant) alongside the 9B. Your workload is close to the ideal case for it: schema-constrained JSON with highly predictable structure ("action": ", field names, braces) gives high draft-acceptance rates. Cost is ~0.5–1GB VRAM — check headroom first (see #4). Configure in the model's load settings or lms load --draft if your CLI version has it.

4. Flash Attention + KV-cache quantization → a third parallel slot
On the 12GB card with the ~6.5GB model, FP16 KV cache at 13000 context is roughly 2GB. Enabling Flash Attention and Q8_0 K/V cache quantization in LM Studio's load settings roughly halves that with negligible quality loss for this task. The reclaimed VRAM could fund either #3, or --context-length 20000 --parallel 3 + MAX_CONCURRENT_LLM = 3 — a +50% throughput bump for the 8-agent roster (per-slot budget ~6600, still above your 3400 floor).

5. Prompt-cache-friendlier prompt assembly
server.py:2712 appends the per-agent persona to the system prompt. LM Studio reuses KV cache by longest common prefix per slot; putting per-agent text at the top of the system message busts the shared prefix immediately, so every one of the 8 agents rotating through 2 slots pays full prompt-processing (~3100 tokens) every turn. Moving the persona line to the top of the user message keeps the large static system prompt as a shared cached prefix. Verify the effect in logs/lm_studio_server.log (it shows per-slot prompt-processing checkpoints).
