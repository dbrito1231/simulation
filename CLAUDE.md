# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A real-time, browser-based AI village simulation. A local LLM (served by LM Studio) acts as the "brain" for each of 8–12 autonomous pixel-art agents that move, talk, trade, gather resources, and collaboratively build structures. It is a proof-of-concept of the LLM-as-brain loop, inspired by Project Sid. Keep it minimal and observable — see [specs/00-overview.md](specs/00-overview.md) for the non-goals.

## Commands

```bash
uv sync                              # install deps (flask, flask-cors, requests)
uv run python simulation/server.py   # start server, then open http://127.0.0.1:5001
```

- LM Studio must be running at `http://localhost:1234` with a model loaded (OpenAI-compatible API). Without it, every agent decision falls back to `rest` and the status dot goes red, but the server stays up.
- **Per-role model routing** (`MODEL_SMART`/`MODEL_FAST` + `model_for_decision()` near the top of `simulation/server.py`): the elder's turns and any turn taken while invention is REQUIRED go to `MODEL_SMART` (blueprint authoring is where model quality matters); all other villager decisions and the background-cognition calls (`lm_complete`) go to `MODEL_FAST`. **Both ids currently point at `qwen/qwen3.5-9b`** — the 2026-07-05 replay benchmark (100 logged prompts, plan Part 6) showed qwen at equal validity but half gemma's `move_to_district` fixation (32% vs 65%), 9 vs 7 distinct actions, and 20/20 valid blueprints, at ~3s/decision more. qwen is a thinking-class model that routes output into `reasoning_content` with empty `content` — the existing extractor handles it. (Historical: `llama-3.2-3b` as a fast tier failed in 2026-07-02 measurements — a 3B can't handle the ~3,100-token prompt; single model on the 12 GB card remains the rule.) Ids must match LM Studio's loaded-model ids (`GET /v1/models`). If a routed id isn't available, `run_agent_decision` prints a notice, flips `_model_routing_enabled` off for the session, and retries with `"local-model"` — so a single-model LM Studio setup still works unchanged. LM Studio target config: **context 13000, parallel 2** (≥3,400/slot with headroom; set via `lms load qwen/qwen3.5-9b --context-length 13000 --parallel 2 -y`).
- **LM Studio context length vs. parallel slots:** the server-authoritative engine's think-worker pool (`MAX_CONCURRENT_LLM` in `simulation/sim_engine.py`, currently **2** — a server-side `ThreadPoolExecutor`, not a client-side queue in `index.html`) runs up to that many decision calls at once, and each one measures **~3,100 prompt tokens** in practice (well above the ~1,500 originally assumed here). LM Studio splits its configured context length across its parallel slots — if `context length ÷ parallel slots` is under ~3,400, expect `"Context size has been exceeded"` errors in bursts under concurrent load. `run_agent_decision()` in `simulation/server.py` detects that specific error and retries once with a slimmed-down prompt (memory/recent-conversations/examples dropped) before giving up the turn to `bad_response_fallback`, and logs the retry distinctly as `context_overflow` in `lm_studio.jsonl`. Set LM Studio's context length to at least `3400 × parallel slots`, or lower `MAX_CONCURRENT_LLM` instead.
- The server (not the browser) serves `index.html` and `sprites.js`, so open `http://127.0.0.1:5001` — do **not** open `index.html` as a file (the spec text predates this; trust the running setup).
- Port is **5001** on purpose (macOS AirPlay squats on 5000).
- Roster size override for experiments: `http://127.0.0.1:5001/?agents=12` (default 8; the builder and elder are always included).
- There is **no test suite, linter, or build step.** `main.py` is an unused stub. Verify changes by running the server and watching the browser + the JSONL logs.

## Architecture

Four code files do all the work, despite [specs/01-architecture.md](specs/01-architecture.md) saying "exactly two", plus one shared data file. The split (and several other deviations) are deliberate and documented in `.cursor/plans/`. **The engine is server-authoritative** (branch `feat/server-authoritative-engine`): the world runs headless in Python; the browser is a thin viewer.

- **[simulation/sim_engine.py](simulation/sim_engine.py)** — the simulation engine (`SimEngine`). Owns ALL world state (the `civilization` dict + `agents` list + `frameTick`), runs the fixed-timestep tick loop (30/s) on a daemon thread, applies decisions via `apply_decision()` (the big world-mutation switch), runs every deterministic system (movement/roads, survival, goals, structure effects, ecology, goods/decay/seasons, rules, memes, the `_maybe_*` backstops), dispatches LLM think jobs to a bounded worker pool (`MAX_CONCURRENT_LLM = 2`, `LLM_MIN_GAP_MS = 250`), and persists the full world to `simulation/state.json` (autosave + graceful-exit flush; `restore_state()` resumes and `setdefault`s new fields for old saves).
- **[simulation/server.py](simulation/server.py)** — Flask app + the cognition side. Serves the viewer (`/`, `/sprites.js`, `/roles.js`, `/districts.js`), the world snapshot (`GET /state`), and controls (`/control/pause|resume|reset`). Hosts the LLM pipeline the engine calls per think job (`run_agent_decision`): builds the system+user prompt, calls LM Studio, extracts JSON from the model output (handles reasoning models that emit empty `content` or code fences), then runs `normalize_decision()` + `role_fallback_action()` to reject invalid/impossible actions and substitute a sensible deterministic one. `SessionLogger` writes per-session JSONL. (`POST /agent/think` still exists as a legacy endpoint but nothing uses it.)
- **[simulation/index.html](simulation/index.html)** — **thin viewer only** (no simulation logic). Polls `GET /state` (~10 Hz) + `GET /districts.js` (slow), renders the snapshot (canvas world, minimap, sidebar, activity/chat panels), and posts the pause/reset controls. Closing the browser does not stop the sim.
- **[simulation/sprites.js](simulation/sprites.js)** — pure Canvas drawing (terrain, zones, agents, structures). Stateless. Custom/blueprint structure types fall back to `drawGenericStructure()`.
- **[simulation/roles.json](simulation/roles.json)** — the **single source of truth for role definitions** (`skill`, `specialty[]`, `preferredProject`, `leader`). The server loads it at startup and derives its role maps (`ROLE_PROJECT`, `RESOURCE_GATHER_ROLES`, `ROLE_PRIMARY_RESOURCE`) from it; it also serves it to the browser as `/roles.js` (a `const ROLES = {…}` global), from which the client derives `ROLE_SKILLS`/`ROLE_PROJECT`/`ROLE_SPECIALTY_RESOURCE`. Edit role data here, never in the two code files.

### Data flow

Server-authoritative: the engine's tick thread advances the world; when an agent's think timer fires, `SimEngine._build_think_payload()` snapshots its context under the lock → `run_agent_decision()` (server.py) prompts LM Studio → JSON decision extracted, validated, normalized (with surfaced `rejection_note`s) → `SimEngine.apply_decision()` mutates the world under the lock. The browser only polls `GET /state` and renders — it holds no simulation state and sends no decisions. All state lives in the engine and persists to `simulation/state.json`.

### The build/civilization pipeline (the heart of the sim, not in the original specs)

`start_project` → agents `collect_resource` → `contribute_resources` → builder `build_structure` once funded → structure placed, `completedProjects++`, civilization level checked. On top of this sits a **blueprint flow**: agents `propose_blueprint` for new structure/resource types, the **elder approves/rejects**, approved blueprints merge into the live registries. The elder (Sage) is the singular leader — it assigns tasks to idle agents and approves blueprints. Earlier the pipeline stalled because the LLM never spontaneously chose `start_project`; the fixes (any role can start projects, elder-driven task assignment, prompt nudges, deterministic fallbacks) live in `.cursor/plans/fix_build_progression.plan.md`.

### Mineflayer-inspired mechanics (feature-flagged constants near the top of sim_engine.py)

Additive systems, each toggled by a module `const`-style flag so behavior can be A/B compared (flags are echoed to the viewer via `/state` `config.flags`). New actions are kept in sync across `DECISION_ACTIONS`/`DECISION_SCHEMA`/`SYSTEM_PROMPT` (server.py), the engine's `apply_decision()` + payload `available_actions` (sim_engine.py), and `ACTION_LABELS` (index.html, display only).

- **`SURVIVAL_ENABLED` (#2)** — agents have `hunger`/`health`; `_update_survival()` (frame-gated by `SURVIVAL_TICK_FRAMES`) drains hunger at `HUNGER_RATE` (0.3/tick — sized so ~3 part-time food producers can feed a roster of 8), then health when starving, and **auto-eats** the first held edible (`_first_edible()` over `EDIBLE_RESOURCES = ["food", "fish"]`, so the fisher self-feeds on his catch — the food consumption sink). At 0 health an agent is `incapacitated` (skips move/think, greyed sprite) until revived. New action **`heal_agent`** restores health (healers heal 2×); any agent can also feed a collapsed neighbour any edible it holds. No permanent death — collapse is recoverable, and both revival paths floor hunger at `REVIVE_HUNGER` (35) so a revived agent doesn't re-collapse seconds later. The food economy is protected deterministically: edibles are **exempt from the resource tax** (the stockpile is never consumed, so taxing them deletes food), `_maybe_force_contribution` leaves each agent an `EDIBLE_RESERVE` (3), a starving agent **auto-receives** an edible from a neighbour within `SHARE_RADIUS` holding above the reserve (`_share_edible_with`), and a foodless agent at hunger ≤ `STARVING_HUNGER` (10) **deterministically seeks the nearest food zone** and gathers (`_maybe_feed_starving`, tick-gated like the other `_maybe_*` backstops — the hunger prompt-nudge is coherence only).
- **`CRAFTING_ENABLED` (#4)** — a `RECIPES` registry produces non-gatherable crafted goods (`planks`/`bricks`/`tools`, `gatherZone:null`) via **`craft_item`** at a station zone. Advanced builds (e.g. the `granary`) require crafted goods → gather→craft→build chain. Agents extend the tree via **`propose_recipe`** / elder **`approve_recipe`**/`reject_recipe`, mirroring the blueprint flow (`pendingRecipes`, `_validate_recipe`).
- **`USE_GOALS` (#1)** — the LLM picks an action; `_goal_for_decision()` turns gather/deliver/craft/build into a persistent `agent["goal"]`, and `_step_goal()` runs it deterministically every `GOAL_STEP_FRAMES` (with a `ttl` cap) **without an LLM call**, consulting the model again only when the goal completes or blocks. Cuts LLM load and improves coherence.
- **`STRUCTURE_EFFECTS_ENABLED`** (`sim_engine.py`, Phase A consequence engine) — one registry (`SEED_STRUCTURE_FUNCTIONS` for seeds, required `function` block on blueprints) declares effects: `produces` (tick-time, logged to `activity.jsonl`), `boosts` (gather/craft), `unlocks` (craft stations), `houses` (population cap). `_tick_structure_effects()` runs every `EFFECT_TICK_FRAMES`; query-time boosts/unlocks/houses go through `_get_structure_function()`. `validate_blueprint()` rejects functionless proposals and duplicate effect vectors (`canonical_effect_vector`); rejections surface via `rejection_note` / `lastBlueprintRejection`. Legacy customs without a stored function get a deterministic default produce. Benchmark: `structure_effect_throughput` in `benchmarks.jsonl`. Saturation (`_type_saturated`) derives soft caps from function params; saturated types are skipped by role defaults, refused by `_start_project_for`, surfaced in prompts, and count toward `_invention_required()` (with `_seed_exhausted` for the granary craft stall).
- **`ECOLOGY_ENABLED`** (`sim_engine.py`, Phase B) — per-district `districtStocks` deplete on gather (`STOCK_DEPLETE_MULTIPLIER=2× yield) and regrow +1 every `ECOLOGY_REGROW_FRAMES` (600); yield scales with stock ratio; zero stock fails with a surfaced `lastGatherRejection` nudge and triggers **`_scarcity_reflex_on_depletion`** (deterministic, no LLM): contribute to matching active terraform → else `start_terraform` in-place → else move to highest-stock district for that resource. **`_craft_input_reflex`** on failed `craft_item` (missing inputs) sets a `craft_gather` goal for the largest deficit (scarcity reflex if depleted); `lastCraftRejection` surfaced in prompts. Stock writes clamp to max; `ecology_scarcity_index` ratios clamp to 1.0. Prompt line `Local resource stocks` + terraform list. **`start_terraform`** action funds `plant_grove` / `clear_field` / `extend_beach` projects; `normalize_decision` in `server.py` infers template ids from district ids, resource names, or fuzzy display names (`lastTerraformRejection` on failure). Completion applies `TERRAFORM_FUNCTIONS` via `_complete_terraform`. Structure `produces` deposit into district stocks when ecology is on. **`_seed_project_from_stockpile`** transfers matching stockpile materials into every newly started build/terraform project (makes abandonment refunds lossless). **`_maybe_abandon_stalled_projects`** cancels district projects with no contribution progress for `PROJECT_ABANDON_THRESHOLD` (10× `STALL_THRESHOLD`, 20× when needs include crafted goods), refunds to stockpile, logs + `lastProjectAbandonment` nudge; after `PROJECT_DEFER_ABANDON_STREAK` (3) consecutive abandonments of the same type, defers that type for `PROJECT_DEFER_COOLDOWN` (20× `STALL_THRESHOLD`). Deferred types are skipped by `_start_project_for`/backstops and do not block `_invention_required()`; deferral clears on successful build or cooldown expiry. **`_start_project_for`** refuses a second active project of the same type (`lastProjectRejection`). **`_maybe_start_approved_custom`** elder-backstops stalled approved customs; on failure tries founding a district of the needed kind, else logs once and backs off (`APPROVED_CUSTOM_BACKOFF_FRAMES`). Squatted stalled projects count as at-capacity in `_kind_at_capacity` so `_maybe_found_district` can open land. Benchmark: `ecology_scarcity_index`. With the flag off, behavior matches Phase A.
- **`GOODS_ENABLED`** (`sim_engine.py`, Phase C physical goods/decay/seasons) — six deterministic mechanics on one slow gate (`GOODS_TICK_FRAMES` ~30s) plus a nightly gate (`DAY_FRAMES` ~7.5 min): (1) **storage** — the Phase A `stores` function effect becomes real: village capacity per resource = `BASE_STORAGE_CAPACITY` (25) + working structures' `stores` entries (the seed granary gains `stores` food 40/fish 20 when the flag is on); (2) **spoilage** — edibles beyond capacity rot `SPOILAGE_RATIO` (25%, min 1) per goods tick, stockpile first then largest holders, never below `EDIBLE_RESERVE`; surfaced via the `lastSpoilage` nudge; (3) **cart** — a seed `cart` recipe (workshop, planks+wood) whose holder carries `COLLECT_CAP + CART_CARRY_BONUS` (query-time `_carry_cap`; the first vehicle); (4) **shelter** — nightly, each *working* house shelters `HOUSE_SHELTER_OCCUPANTS` (2, nearest agents first); the unsheltered lose `SHELTER_HUNGER_PENALTY` hunger (floored at `SHELTER_HUNGER_FLOOR` — a nudge, never punitive), surfaced via `lastShelterNote`; (5) **decay & repair** — structures carry `condition` (100 at build, −`STRUCTURE_DECAY_PER_GOODS_TICK`/tick, sized as the consumer for the ~30 builds/hour sprawl); below `STRUCTURE_DISREPAIR_THRESHOLD` (30) they stop producing/boosting/housing/unlocking (`_working_structure_count`; `_type_saturated` deliberately still counts totals), at 0 they collapse to a ruin; the **`repair_structure`** action (synced across `DECISION_ACTIONS`/`SYSTEM_PROMPT` in server.py, engine `apply_decision`, `ACTION_LABELS` in index.html) restores +`REPAIR_CONDITION_RESTORE` for 1 unit of the structure's primary material and rebuilds a ruin for **half** its original needs — the deterministic escape; rare disasters (`DISASTER_PROB` per goods tick) damage a random structure, logged dramatically; refusals surface via `lastRepairRejection` + an in-district disrepair nudge; (6) **seasons** — a frameTick-derived four-season clock (`SEASON_FRAMES` ~30 min each) multiplies ecology regrowth (`SEASON_REGROW_MULT`: spring 2×, winter 0×) and adds one `Season:` prompt line rendered only when the flag is on (flag-off prompts stay byte-identical). Benchmarks: `storage_utilization` (with spoiled-per-period), `structure_condition` (avg + ruins/disrepair counts), `season_turn`, `disaster_damage`, `structure_repaired`. Structures from pre-Phase-C saves need no migration (every condition read defaults to 100). With the flag off, behavior matches Phase B exactly.

- **`TECH_TREE_ENABLED`** (`sim_engine.py`, Phase D tech tiers & eras) — every structure/recipe/blueprint carries a `tier` (seeds = 1; `forge`, `granary`, `cart` = 2); `_type_tier_locked` gates a tier-N recipe/blueprint behind a **built, working** tier-(N-1) station with a surfaced reason ("requires a forge (tier 2) first") through `validate_blueprint`/`propose_recipe`/`craft_item` — the Forge itself is a normal tier-1 build, so the gate always has a deterministic escape. `_village_tech_tier`/`_craft_station_unlocked` are tier-aware for query-time unlocks. **Era** replaces the vanity `level` in prompts/UI (`_current_era_name`, computed from capability set — has a working tier-2 station, has a vehicle, …; `level` stays in state for back-compat but isn't surfaced), logging dramatic transitions ("The village enters the Forge Era!"). The **wagon** is the Phase C cart's tier-2 upgrade (crafted at the Forge, consumes a cart) — a mobile structure assigned to its holder, boosting both carry cap and movement speed at query time (`_carry_cap`, `_vehicle_speed_mult`) — the "reachable, never named into existence" answer to vehicles. A model-experiment hook (`INVENTION_TEMPERATURE`/`INVENTION_MAX_TOKENS` near `MODEL_SMART` in server.py) lets invention-only calls override temperature/max_tokens; both default to `None` (current behavior) until a replay experiment flips them. **Invention council** (diegetic LLM-council pattern): when `_maybe_invention_backstop` fires with 2+ idle villagers, it fans 2–3 of them out to parallel invention-only turns (`INVENTION_COUNCIL_SIZE`, replacing their normal turn — no added LLM volume) that each independently `propose_blueprint`; when 2+ proposals are pending, the elder's next turn is a **comparative** judgment (needs/function summary listed side by side, approve-one + reject-the-rest-with-reasons in one call), logged as a village event ("Elder Sage chose the Windmill over fish_smokery: ..."). `civilization["councilLog"]`/`councilActive` are capped, persisted, and served over `GET /state`; index.html renders them in a thin, stateless Council sidebar panel (winner highlighted, losers greyed with reasons, collapsible history) plus a live "Council in session" banner — no simulation logic moved to the browser. `restore_state()` `setdefault`s `era`/`eraIndex`/`councilActive`/`councilLog` and merges the new seed registry entries (Forge, wagon) into old registries, so pre-Phase-D saves load and can build the new tier without migration. With the flag off, behavior matches Phase C exactly.

### Invention pipeline safeguards (added 2026-07-03 after a 12h deadlock)

`MAX_APPROVED_CUSTOM` (15) used to deadlock the sim: once hit, `validate_blueprint` rejected every proposal (1,344 silent rejections in one session) while `_invention_required()` kept demanding one. Now: (1) `_maybe_retire_blueprint` archives the oldest **built** custom blueprint to keep an approval slot free (structures carry their own name/visualStyle, so nothing on the map changes; a retired id may legitimately be re-invented later); (2) `_invention_required()` returns False at the cap as a safety net; (3) blueprint rejections are no longer silent — `normalize_decision` attaches `rejection_note`, the engine stores it on the agent, and the next prompt says why (`lastBlueprintRejection` nudge); (4) `_maybe_invention_backstop` flags its target for a one-shot **invention-only turn** (`agent["inventionTurn"]` → payload `invention_only` → `build_invention_prompt` in server.py, a slim proposal-only prompt routed to `MODEL_SMART`), and after `INVENTION_ELDER_TAKEOVER` fruitless delegations (or when no villager is free) the elder takes that turn himself; (5) the "use start_project now" nudge is suppressed while invention is required (it contradicted the gate). Conversation balance: the elder can't re-task an agent within `ELDER_RETASK_COOLDOWN_FRAMES`, and a long-silent agent with neighbors gets a `talk_to_nearby` nudge (`SOCIAL_SILENCE_FRAMES`, tracked via `lastSpokeFrame`).

### Sage-priority emergency (absolute, deterministic — sim_engine.py)

The elder Sage's survival overrides everything. `_sage_emergency()` returns the agent to revive when Sage is collapsed or below `SAGE_CRITICAL_HEALTH` (**30**) — the **healer first if she is also down** (she's the key to saving Sage), otherwise Sage. Only a scoped responder set rushes in: `_sage_responders(target)` is the **healer plus the single nearest able agent** (≤2 villagers), so everyone else keeps working (the original "all villagers rush" rule was overkill — it caused ~270 rushes/session). The tick loop runs `_rush_to_heal()` for just those responders (clearing their `goal`, skipping LLM), and `_think_job` has an **in-flight guard** that discards a responder's LLM decision that lands mid-emergency — so no task, goal, or model output can take precedence for them (enforcement is deterministic; the `SYSTEM_PROMPT` rule #15 + a responder-only nudge are only for coherence). Collapse stays recoverable: assisted `heal_agent` revives fast, and `COLLAPSE_REGEN` self-revives an unattended agent slowly (`COLLAPSE_REVIVE_HEALTH`) so no one is permanently stuck. A `CRAFT_STALL_THRESHOLD` nudge pushes the village to build a workshop and craft when the chain has stalled.

## Specs vs. reality

The `specs/` directory is the original 6-gate build plan and is partly **superseded**. Where it conflicts with the running code, prefer the code and the plans in `.cursor/plans/`:

- "Exactly two files" → four (sprites.js added, then the engine port split sim_engine.py out of the browser).
- "Exactly 12 agents" → default 8, URL-overridable.
- Port 5000 → 5001; browser opens the server URL, not a local file.
- 4 fixed projects / 3 fixed resources → dynamic registries extended at runtime by the blueprint flow.

[simulation/ISSUES.md](simulation/ISSUES.md) is a diagnostic of the *previously broken* state — useful context, but most of what it flags is addressed by the fix plans.

[docs/project-sid-parity-roadmap.md](docs/project-sid-parity-roadmap.md) maps the gap between this codebase and the Project Sid paper (`docs/2024-10-31.pdf`) and lays out a tiered roadmap (memory, emergent roles, voting, memes, PIANO) for closing it in 2D.

## Logs

Each server run writes to `simulation/logs/<timestamp>/` (gitignored): `activity.jsonl` (world events), `conversation.jsonl` (agent dialogue), `lm_studio.jsonl` (full LLM request/response/decision per call). These are the primary debugging surface — read `lm_studio.jsonl` to see what the model actually returned and which fallback fired.

`simulation/logs/lm_studio_server.log`, if present, is LM Studio's *own* server log (not written by this app) — useful for diagnosing model-side issues like context-window/parallel-slot sizing, since it shows token usage and per-slot context checkpoints that `lm_studio.jsonl` doesn't.
